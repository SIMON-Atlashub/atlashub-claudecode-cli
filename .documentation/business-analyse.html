Introduction

Le workflow Business Analyse est specialise pour la specification fonctionnelle et technique. Il combine la methodologie BABOK v3 avec une approche systematique pour produire des specifications completes et autonomes. Le BA ne code jamais - il produit uniquement des documents de specification.

The Business Analyse workflow is specialized for functional and technical specification. It combines BABOK v3 methodology with a systematic approach to produce complete and autonomous specifications. The BA never codes - it only produces specification documents.

⚠
Principe fondamental
Core Principle

LE BA NE CODE JAMAIS. Il produit des specifications suffisamment detaillees pour qu'un developpeur puisse implementer de maniere autonome, sans poser de questions.

THE BA NEVER CODES. It produces specifications detailed enough for a developer to implement autonomously, without asking questions.

💡
Questions adaptatives
Adaptive Questions

Le nombre de questions s'adapte a la complexite : 6-8 pour un CRUD simple, jusqu'a 30+ pour une feature critique. Plus besoin de repondre a 44 questions pour un simple formulaire !

The number of questions adapts to complexity: 6-8 for a simple CRUD, up to 30+ for a critical feature. No more answering 44 questions for a simple form!

🎯
BA Proactif
Proactive BA

Le BA propose des fonctionnalites que l'utilisateur n'a pas pensees, dessine des wireframes ASCII et challenge agressivement les besoins vagues.

The BA suggests features the user didn't think of, draws ASCII wireframes and aggressively challenges vague requirements.

🌐

Optimized for One-Shot

Le prompt de developpement genere est en anglais et contient tout le necessaire pour que Claude Code implemente la feature sans poser de questions.

The generated development prompt is in English and contains everything needed for Claude Code to implement the feature without asking questions.

Output : config.json (configuration projet), glossary.md (glossaire metier), .claudeignore (exclusions), context.md (contexte applicatif avec stakeholders et modules).

2. Discover OPUS + ultrathink

The BA uses OPUS + ultrathink to aggressively challenge assumptions, identify unstated requirements, propose related features the user hasn't thought of, and draw ASCII wireframes to validate the interface vision. This is the most critical phase as it defines scope and business rules.

# Business Analyse - Discover

Senior BA expert in elicitation. OPUS + ULTRATHINK mode mandatory.

## Model Requirement
**This phase REQUIRES OPUS model** for exhaustive elicitation.

## Arguments
/business-analyse:discover [module] "needs description"
- module: Name of the concerned module
- description: Initial description of the need

## ULTRATHINK Mode
This phase uses OPUS + deep thinking:
- Question each hypothesis
- Look for contradictions
- Identify what's left unsaid
- Explore edge cases
- Validate priorities

## Adaptive Questions by Complexity

| Complexity | Questions | Priority areas |
|------------|-----------|----------------|
| Simple CRUD | 6-8 | Context, Data, Permissions |
| Standard | 12-15 | + Business rules, UI |
| Complex | 20-25 | + Edge cases, Integrations |
| Critical | 30+ | All areas |

### ESSENTIAL questions (always asked - 6 questions)
1. What problem does this feature solve?
2. Who uses it? (main roles)
3. What data is manipulated? (CRUD entities)
4. What permissions? (who can do what)
5. Scope: what is IN and OUT?
6. Are there specific business rules?

### CONDITIONAL questions (based on complexity)
- If >= Standard: Main flow, Validations, Messages, Audit trail
- If >= Complex: Integrations, Concurrency, Volume, Edge cases
- If = Critical: GDPR, Legal, Performance, Irreversible operations

## Output
Creates 1-discovery.md with:
- Executive Summary, Strategic Context, Users, Scope
- Data, Business Flows, Business Rules, Security, Risks
- Open Questions, Next Steps

3. Analyse ultrathink

Fonctionnalite : Cette phase transforme les informations brutes de la discovery en analyse metier structuree. Elle scanne d'abord les specifications existantes pour eviter les duplications, puis modelise les processus metier en diagrammes Mermaid (AS-IS/TO-BE), cree le modele conceptuel de donnees (ER diagram), catalogue exhaustivement les regles metier avec leur matrice de tracabilite, et analyse les impacts sur les systemes/donnees/utilisateurs existants.

Functionality: This phase transforms raw discovery information into structured business analysis. It first scans existing specifications to avoid duplications, then models business processes in Mermaid diagrams (AS-IS/TO-BE), creates the conceptual data model (ER diagram), exhaustively catalogs business rules with their traceability matrix, and analyzes impacts on existing systems/data/users.

Output : 2-business-requirements.md (BRD) contenant : objectifs metier, analyse des stakeholders, exigences fonctionnelles/non-fonctionnelles, modele de donnees conceptuel, diagrammes de processus Mermaid, catalogue des regles metier, analyse d'impact, risques et dependances.

Output: 2-business-requirements.md (BRD) containing: business objectives, stakeholder analysis, functional/non-functional requirements, conceptual data model, Mermaid process diagrams, business rules catalog, impact analysis, risks and dependencies.

Exemple : State Machine Pattern

Example: State Machine Pattern

## State Machine: [Entity] Status

### States
| State | Description | Entry Actions | Exit Actions |
|-------|-------------|---------------|--------------|
| DRAFT | Initial state | Set createdAt | Log transition |
| ACTIVE | Published | Notify users | None |
| CLOSED | Terminated | Archive data | Cleanup cache |

### Transitions Matrix
| From → To | Trigger | Condition | Side Effects |
|-----------|---------|-----------|--------------|
| DRAFT→ACTIVE | publish() | isValid() | sendNotification() |
| ACTIVE→CLOSED | close() | canClose() | archiveData() |

### Illegal Transitions
| Transition | Reason | Error Code |
|------------|--------|------------|
| CLOSED→ACTIVE | Archived data cannot be reactivated | ERR_ILLEGAL_TRANSITION |
| DRAFT→CLOSED | Must be published first | ERR_INVALID_STATE |

Exemple : Error Recovery Pattern

Example: Error Recovery Pattern

## Error Recovery: Conflict Handling (409)

### Conflict Detection
- Compare entity version/ETag before save
- If mismatch: 409 Conflict response

### User Resolution Options
| Option | Behavior | UI Action |
|--------|----------|-----------|
| Force overwrite | My changes win | "Save anyway" button |
| Merge changes | Show diff, let user choose | Side-by-side view |
| Discard mine | Their changes win | "Reload" button |

### Retry Strategy (Network Errors)
| Attempt | Delay | Max |
|---------|-------|-----|
| 1 | 1s | 3 attempts |
| 2 | 2s | exponential backoff |
| 3 | 4s | show "Retry manually" |

Hierarchie des Contraintes (v10.0)

Constraint Hierarchy (v10.0)

Le DEV-PROMPT utilise RFC 2119 pour classifier les regles metier selon leur criticite :

The DEV-PROMPT uses RFC 2119 to classify business rules by criticality:

Niveau Level Signification Meaning Implementation Implementation
MUST Obligatoire, echec = bug Mandatory, failure = bug Validation stricte, tests obligatoires Strict validation, required tests
SHOULD Recommande, exceptions documentees Recommended, documented exceptions Warning si ignore, log raison Warning if skipped, log reason
MAY Optionnel, nice-to-have Optional, nice-to-have Implementer si temps disponible Implement if time allows 
## Constraint Hierarchy Example

### MUST (Mandatory - Zero tolerance)
- [ ] User email MUST be unique (throw 409 if duplicate)
- [ ] Password MUST be hashed with bcrypt (cost factor 12)
- [ ] All API endpoints MUST validate authentication token

### SHOULD (Recommended - Document exceptions)
- [ ] Response time SHOULD be under 200ms for list endpoints
- [ ] Error messages SHOULD be user-friendly (not technical)
- [ ] Form validation SHOULD happen client-side first

### MAY (Optional - Nice-to-have)
- [ ] User MAY set preferred language
- [ ] Dashboard MAY show usage statistics
- [ ] Export MAY support CSV and Excel formats

Bug Tracking Bug Tracking

Fonctionnalite : Cette commande permet de documenter et specifier la resolution d'un bug de maniere structuree. Le BA collecte les informations de reproduction (severite, etapes, environnement), analyse systematiquement les causes probables (spec manquante, ambigue, regression, technique), identifie les composants impactes, et specifie le comportement correct attendu. Le bug est trace vers les specifications existantes qui seront mises a jour si necessaire.

Functionality: This command documents and specifies bug resolution in a structured manner. The BA collects reproduction information (severity, steps, environment), systematically analyzes probable causes (missing spec, ambiguous, regression, technical), identifies impacted components, and specifies the expected correct behavior. The bug is traced to existing specifications that will be updated if needed.

Output : BUG-XXX.md dans tracking/bugs/ contenant : metadonnees (severite, reproductibilite, statut), description du comportement observe vs attendu, etapes de reproduction, analyse des causes probables, impact utilisateur, composants impactes, specification du fix avec criteres d'acceptation Gherkin, tests de non-regression suggeres.

Output: BUG-XXX.md in tracking/bugs/ containing: metadata (severity, reproducibility, status), observed vs expected behavior description, reproduction steps, probable cause analysis, user impact, impacted components, fix specification with Gherkin acceptance criteria, suggested non-regression tests.

# Business Analyse - Bug

Bug documentation and specification with root cause analysis.

## Arguments
/business-analyse:bug "bug description"
- description: Clear description of the bug

## Bug Report Structure

### 1. Identification
- Bug ID (BUG-XXX)
- Severity (Critical, High, Medium, Low)
- Priority
- Affected version
- Reporter

### 2. Reproduction
- Steps to reproduce
- Environment details
- Expected vs Actual behavior
- Screenshots/logs if available

### 3. Root Cause Analysis
- Component affected
- Code location (if known)
- Root cause hypothesis
- Impact analysis

### 4. Solution Specification
- Proposed fix description
- Files to modify
- Risk assessment
- Rollback plan

### 5. Validation Criteria
Gherkin format:
Given [initial state]
When [action taken]
Then [expected result]

## Output
Creates bug report in:
.business-analyse/applications/.../tracking/bugs/BUG-XXX.md
